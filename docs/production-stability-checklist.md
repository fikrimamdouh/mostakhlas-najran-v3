# Production Stability Checklist — Mostakhlas Najran

الغرض: إثبات الاستقرار قبل الإغلاق، بدون تعديل التصميم أو المعادلات أو تدفق العمل.

> لا يتم اعتبار المشروع جاهزًا للإغلاق إلا بعد تنفيذ هذه الاختبارات على بيئة الإنتاج أو staging مطابق للإنتاج.

---

## 1) مستخدم عادي

### السيناريو
- تسجيل دخول بحساب مستخدم عادي مرتبط بمستشفى واحدة.
- فتح `settings_main.html`.
- حفظ إعدادات المستخلص: الشهر، السنة، رقم المستخلص، تاريخ البداية، تاريخ النهاية.
- فتح صفحة الحضور.

### التحقق
- [ ] اسم المستشفى الظاهر مطابق لمستشفى المستخدم.
- [ ] الشركة والعقد مطابقان لمستشفى المستخدم.
- [ ] مدة المستخلص لا ترجع لقيمة قديمة بعد فتح الحضور.
- [ ] لا تظهر بيانات مستشفى آخر.
- [ ] لا يظهر `PUSH` تلقائي لبيانات تشغيلية بعد تعديل حضور أو أداء.
- [ ] بيانات الحضور تحفظ محليًا بعد التعديل.

### Console / Network
- [ ] لا توجد أخطاء JavaScript جديدة.
- [ ] طلبات `GET /api/hospital-storage` مربوطة بالمستشفى الصحيح.
- [ ] لا يوجد `PUT /api/hospital-storage?hospital=...`.

---

## 2) مراجع فقط — Review Only

### السيناريو
- تسجيل دخول بحساب له مستشفيات مراجعة فقط.
- فتح مستشفى مراجعة.
- فتح صفحة إعدادات/حضور/اعتماد حسب الصلاحية.

### التحقق
- [ ] تظهر حالة مراجعة فقط أو log واضح `REVIEW ONLY`.
- [ ] البيانات المعروضة من السيرفر فقط.
- [ ] لا يتم رفع بيانات عند التصفح.
- [ ] لا يتم رفع بيانات عند الخروج.
- [ ] لا يتم تعديل بيانات تشغيلية محلية بشكل مؤثر.

### Network
- [ ] لا يوجد `PUT /api/hospital-storage`.
- [ ] لا يوجد `POST` أو `PUT` تشغيلي غير مصرح.
- [ ] يسمح فقط بطلبات القراءة اللازمة.

---

## 3) أدمن / تبديل مستشفيات المراجعة

### السيناريو
- تسجيل دخول أدمن.
- التبديل بين:
  - مستشفى بدر الجنوب العام
  - مستشفى حبونا العام
  - مستشفى يدمة العام

### التحقق
- [ ] اسم المستشفى يتغير فورًا بعد التبديل.
- [ ] بيانات العقد تتغير أو تثبت حسب العقد الصحيح للموقع.
- [ ] بيانات المستخلص تتغير حسب الموقع الحالي.
- [ ] لا تبقى بيانات المستشفى السابق ظاهرة.
- [ ] الحضور لا يعرض أسماء أو أيام من مستشفى سابق.

### Network / Storage
- [ ] طلبات القراءة تذهب للمستشفى المختار فقط.
- [ ] لا يوجد رفع حضور من الأدمن إلا إذا كان هذا مسموحًا صراحة عبر صلاحية تحرير الحضور.

---

## 4) API Permissions

### submitted-extracts
- [ ] `PUT /api/submitted-extracts/:id` لا يقبل `hospitalName` من body.
- [ ] `PUT /api/submitted-extracts/:id` لا يقبل `companyName` من body.
- [ ] `PUT /api/submitted-extracts/:id` لا يقبل `contractNumber` من body.
- [ ] المستخدم لا يستطيع تعديل مستخلص ليس له.
- [ ] إعادة التقديم مسموحة فقط لحالات `needs_revision` أو `rejected`.

### hospital-storage
- [ ] المستخدم العادي يكتب فقط على مستشفاه.
- [ ] المراجع لا يكتب على مستشفى مراجعة.
- [ ] وجود `hospital=` في طلب PUT مرفوض.

### contract supervisor
- [ ] مشرف العقد لا يرى خارج نطاق شركته.
- [ ] لا يستطيع تنفيذ عمليات اعتماد خارج صلاحياته.

---

## 5) localStorage قديم

### السيناريو
- فتح النظام من جهاز يحتوي بيانات قديمة في localStorage.
- تسجيل دخول كمستخدم مستشفى مختلفة عن البيانات القديمة.
- فتح settings ثم attendance.

### التحقق
- [ ] لا تظهر بيانات المستشفى القديم.
- [ ] لا ترجع مدة مستخلص قديمة بعد الحفظ.
- [ ] لا يتم رفع بيانات قديمة للسحابة تلقائيًا.
- [ ] إذا ظهرت بيانات قديمة كقالب لأول مستخلص جديد، لا يتم رفعها إلا بعد تعديل/رفع صريح.

---

## 6) reload بعد الحفظ

### السيناريو
- حفظ إعدادات المستخلص.
- عمل reload للصفحة.
- فتح الحضور.

### التحقق
- [ ] الشهر والسنة ورقم المستخلص ثابتة.
- [ ] تاريخ البداية والنهاية ثابتان.
- [ ] مدة المستخلص لا ترجع.
- [ ] لا يوجد overwrite من السحابة فوق المحلي الأحدث بدون قرار واضح.

---

## 7) جهازان لنفس المستشفى

### السيناريو
- جهاز 1 يفتح نفس المستشفى ويعدل بيانات تشغيلية محليًا.
- جهاز 2 يفتح نفس المستشفى.
- جهاز 1 يرفع صراحة.
- جهاز 2 يعيد فتح الصفحة.

### التحقق
- [ ] لا يحدث خلط بين مستشفيات.
- [ ] لا يحدث overwrite صامت لبيانات أحدث.
- [ ] المراجعة تعرض snapshot المرفوع لا حالة محلية عشوائية.

---

## 8) الأداء

### التحقق
- [ ] فتح settings_main لا يتجاوز زمنًا غير طبيعي.
- [ ] فتح attendance لا ينفذ API calls متكررة بلا داعٍ.
- [ ] لا يوجد loop مستمر في Console.
- [ ] لا يوجد `PUSH` تلقائي متكرر للبيانات التشغيلية.
- [ ] Vercel data transfer يبدأ في الانخفاض بعد تطبيق cache bust يدويًا لـ cloud-sync.
